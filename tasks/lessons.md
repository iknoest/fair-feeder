# Lessons Learned

After every correction, this file is updated to prevent repeating mistakes.
Review this file at the start of each session.

---

| # | Lesson | Context |
|---|--------|---------|
| 1 | `.ipynb` files cannot be edited with editor tools — use a Python script to modify the JSON | Tried to use `multi_replace_file_content` on notebook, got blocked |
| 2 | Never hardcode camera credentials in fallback defaults — use `<PLACEHOLDER>` patterns | Tapo password was committed in source via `os.getenv('TAPO_PASS', 'real_password')` |
| 3 | `create_pullpoint_manager()` requires `subscription_lost_callback` keyword argument | ONVIF library TypeError when `subscription_lost_callback` was positional |
| 4 | Tapo ONVIF sends motion events in bursts with 1-3s gaps — never use instantaneous `motion_detected` flag for stop decisions | Recording kept stopping and restarting during continuous motion |
| 5 | Windows PowerShell redirect (`>`) produces UTF-16LE files — use Python `open()` with explicit `encoding='utf-8'` instead | `view_file` tool rejected UTF-16LE text files |
| 6 | Synchronous external commands (like file uploads) block the Python event loop and cause dropped frames | Video uploads prevented the script from resetting for the next capture; used `subprocess.Popen` instead for fire-and-forget sync |
| 7 | Cross-platform hardcoded paths break deployment — use `platform.system()` checks to conditionally set paths | Windows uses drive letters (`H:\`), Pi uses Linux mount points (`/home/pi5/...`) |
| 8 | **ai-edge-litert is newer (2024) than tflite_runtime — APIs are incompatible** | Tried `interpreter.invoke([input_data])` but ai-edge-litert uses `interpreter.invoke()` with pre-set tensor. Problem: requires `allocate_tensors()` and `set_tensor()` before invoke(). Solution: test inference API thoroughly on target platform. |
| 9 | **Test all dependencies on target platform BEFORE deploying** | Installed packages on Windows, but `mediapipe` + `tensorflow` unavailable on Pi ARM. Use lightweight alternatives: `test_motion_pi.py` (pure OpenCV MOG2) instead of complex ML stacks. |
| 10 | **Motion detection works on Pi 5, cat detection is fragile** | `test_motion_pi.py` (MOG2 motion detection) succeeds. `motion_recorder.py` (TFLite inference) crashes. For production: prioritize simplicity. Use `test_motion_pi.py` (all motion saved) OR fix TFLite API in vision/detector.py. |
| 11 | **EfficientDet is bad at extreme angles, use YOLOv8n** | Testing `efficientdet_lite2` proved it hallucinated objects ("oven", "sink") when the camera is placed on the floor pointing at the cat bowl. YOLOv8n correctly identifies objects via the ultralytics standard pipeline, even picking up partial cat bodies from a ground-level view with a lower confidence threshold (0.10). |
| 12 | **Always verify the remote environment BEFORE assuming failure** | Ignored the fact that `ai-edge-litert` (the modern `tflite_runtime` replacement) was already successfully installed on the Pi 5. Assumed edge inference was dead instead of checking `pip list` or evaluating the installed tools properly. |
| 13 | **If an SDK fails (like on ARM64), pivot immediately to REST APIs** | The official `infisical-sdk` failed to install on the Pi 5 because it lacked pre-built ARM binaries. Instead of giving up or waiting for a prompt, use standard `requests` to securely query the Infisical REST API (Universal Auth). Fallbacks are better than roadblocks. |
| 14 | **GitHub Actions secrets must be explicitly listed in the `env:` block of the step that reads them** | Adding a secret to repo settings does NOT auto-inject it into the environment. `os.environ.get('SECRET', '')` silently returns empty string if the secret is omitted from the workflow `env:` block. Symptom: notebook runs but all API calls fail silently. Fix: list every secret the notebook reads under `env:` in the papermill step. |
| 15 | **Service accounts cannot CREATE files in a personal Google Drive (zero quota)** | `files().create()` raises `HttpError 403 storageQuotaExceeded` because service accounts have no storage quota on personal Drive. `files().update()` on an existing file works fine (uses the file owner's quota). Fix: pre-create the file manually in Drive UI, share it with the service account (Editor), then only ever call `update()`. Wrap `create()` in try/except so CI doesn't crash on first run before the file exists. |
| 16 | **Use `tqdm.auto` not `tqdm.notebook` in notebooks that may run in CI** | `tqdm.notebook` requires `ipywidgets.IntProgress` (IProgress). In papermill/CI there is no Jupyter widget server → `IProgress` is `None` → `ImportError` at import time. `tqdm.auto` detects the environment and falls back to plain text automatically, working in both Colab (widgets) and CI (text) without extra packages. |
| 17 | **Audit every cell for Colab-specific imports when adding CI support** | `from google.colab import userdata`, `from infisical_sdk import ...`, and `drive.mount('/content/drive')` all crash in CI. Every one must be inside `if not RUNNING_IN_CI:`. Do a full cell-by-cell audit when porting a Colab notebook to run under papermill. |
| 18 | **A scheduled or re-triggered workflow may check out old code if the push hasn't landed yet** | If a workflow runs just before a fix is pushed, GitHub Actions checks out the pre-fix commit. Symptom: error traceback shows wrong line numbers. Fix: confirm the fix is on main, then re-trigger the workflow manually. |
| 19 | **`cv2.VideoWriter` FPS must match the stream's actual delivery rate, not a hardcoded constant** | Hardcoding `VIDEO_FPS = 15` as the writer FPS causes sped-up playback when the RTSP stream delivers fewer real frames per second. Each frame is stamped `1/15 s` apart, so a 2.5-minute real event with only ~880 frames plays back in ~58s. Fix: read `cap.get(cv2.CAP_PROP_FPS)` after opening the stream and use that as the writer FPS; then remux with ffmpeg `setpts` if actual vs declared rate diverges by >20%. |
| 20 | **Git remote operations require auth on Pi** | Running `git pull` directly on the Pi paused and failed waiting for GitHub credentials. Workaround: run `git archive` on the authenticated Windows machine, transfer the `.zip` via SCP, and extract it on the Pi. |
| 21 | **Bot listener must use same auth method as startup alert** | The TelegramCommandListener failed to start because it only checked `.env` for credentials, while the startup message correctly checked Infisical REST API. Ensure all components use the unified `get_telegram_credentials()` helper. |
| 22 | **Avoid rclone bisync for constant daemon uploads** | `rclone bisync` creates deadlocks (.lck files) if interrupted, locking up the sync state forever. For a simple recording daemon, use one-way `rclone copy` to instantly upload new videos instantly, and use a separate daily cron job with `find -mtime` to safely purge old space locally. |
| 23 | **GitHub Actions apt-get install failures (404 Not Found)** | `sudo apt-get install` without `sudo apt-get update` first can fail if the Ubuntu package mirror repository cache is stale. Always run `sudo apt-get update` before installing system dependencies via apt in workflows. |
| 24 | **"Fixed" commits for Drive uploads must be verified by reading the actual Drive file** | `98c3a47` was committed as "properly append to GDrive csv and upload output video" but the Drive folder still shows no annotated video and the CSV has not accumulated. Code that runs without error does not mean the file appeared in Drive. Always verify by checking the actual target folder contents after a CI run. |
| 25 | **The detection timeline and FeedingTracker analytics are separate pipelines — a working timeline does not mean analytics are working** | The timeline chart is generated from raw detection frames directly. FeedingTracker (Phase 2) is a separate processing path. Both can produce opposite results from the same cache. When the timeline shows kibble and cat-at-bowl data but the summary reports 0, the analytics pipeline is disconnected from the data — not a detection problem. Debug Phase 2 independently from the chart. |
| 26 | **When clips are merged, ALL Phase 3 outputs must reference the merged output path, not the per-clip path** | After stitching clips into a merged video, Phase 3 still sent the short unmerged clip's annotated video to Telegram. The merged video was correctly analyzed (Phase 1 ran on it), but Phase 3 derived the annotated video filename from the original per-clip variable. When a merge occurs, Phase 3 must branch on the merged output path for both the Telegram video send and the Drive upload. |
| 27 | **Never mark a multi-step fix as complete unless every component is independently verified** | Issues #31–34 all surfaced after commits that claimed to fix them. The pattern: fix one layer (e.g. auth, quota error) without confirming the end-to-end result (file in Drive, CSV row added, correct video sent). Each observable output needs its own verification step before marking done. |
| 28 | **Stitching clips without checking the gap creates logically invalid merged events** | Two clips in the feeding window can be 5+ minutes apart (separate events). Stitching them blindly produces a single FeedingTracker run that spans two unrelated feeding sessions, making kibble attribution meaningless. Always compute the gap between clip end and next clip start — only stitch if gap ≤ 10 seconds; otherwise treat as separate events with independent analysis. |
| 29 | **Don't fight the SA quota — use a different delivery channel or runtime** | Trying to work around SA's zero storage quota on personal Drive (pre-create placeholders, user OAuth tokens, rclone configs) is fragile and creates ongoing maintenance. Better: use Telegram for daily delivery (already working), Colab for archiving (Drive mounted as user), and Roboflow for flagged frames. Each tool uses the auth model that fits it. |
| 30 | **Close the model improvement loop with auto-flagging, not manual frame extraction** | Reviewing annotated videos by eye and manually extracting bad frames for retraining is tedious and doesn't scale. Instead: auto-flag suspicious detections (low-confidence, single-frame blips, conflicting classes, kibble count jumps) from the existing detection cache, upload flagged frames to Roboflow with tags. Human reviews only the flagged subset in Roboflow's labeling UI. |
| 31 | **Notebook cells injected via Python scripts must guard against Windows CRLF** | When building cell source arrays programmatically on Windows, always `.replace('\r', '')` and use `newline='\n'` when writing the JSON. CRLF in Jupyter cell source breaks the IPython lexer in papermill/CI. |
| 32 | **One-shot notebook patch scripts (`fix_*.py`) should not be committed to the repo** | These scripts modify `.ipynb` JSON once and are never needed again. Committing them clutters the repo and confuses contributors. Keep them local or delete after use. Only commit reusable modules (e.g., `flagging.py`, `roboflow_upload.py`) and their tests. |
| 33 | **When writing shared modules used by both CI and Colab, handle both old and new data formats** | Historical detection caches use `{class, confidence, box}` but new CI cache uses `{class_name, conf, x1/y1/x2/y2}`. Shared modules must detect and handle both formats or batch_review.ipynb breaks on historical data. Use accessor helpers (`_det_cls`, `_det_conf`, `_det_box`). |
| 34 | **`FLAG_KIBBLE_JUMP=5` generates excessive noise — use 15 for meaningful kibble-jump flags** | At threshold 5, normal occlusion (cat blocking kibble view) floods Roboflow with kibble-jump frames that aren't labeling errors. Threshold 15 catches real model failures (count changes of 16+) while filtering routine occlusion noise. Verified across 19 feeding-window videos: 465→263 flagged frames, cleaner signal. |
| 35 | **Re-evaluate augmentation when dataset balance changes between versions** | v13 used `copy_paste=0.3` because Kibble was underrepresented. In v14 Kibble became the largest class (4015 annotations) — copy-paste would amplify the wrong class and worsen imbalance. Always check class counts before blindly reusing the same augmentation settings. |
| 36 | **Model capacity must match dataset size — don't scale up the model, scale up the data** | With 775 training images, YOLOv11s (9.4M params) is already tight on the ~1000-2000 images/million-params guideline. YOLOv11m (20M params) would overfit. The bottleneck for improving mAP is more diverse training data (data flywheel), not more parameters. |
| 37 | **`roboflow_upload.py` `_det_box()` must handle all three detection cache formats** | Historical pkl files use `bbox` key, old morning_report uses `box`, new CI uses `x1/y1/x2/y2`. Missing one format causes silent upload failures (exception swallowed by broad `except`). Check order: `x1` → `bbox` → `box`. Same fix was already in `flagging.py` but was missed in `roboflow_upload.py`. |
| 38 | **Infisical pip package renamed from `infisical-sdk` to `infisicalsdk`** | The old package name is deprecated. Import stays `from infisical_sdk import InfisicalSDKClient`. Update all `pip install` lines in notebooks. |
| 39 | **OCR character replacements must consider multi-character patterns first** | EasyOCR reads Tapo's `:1` as `\|:`. Replacing `\|` → `1` alone gives `6:201:0` (extra colon). Must replace `\|:` → `:1` first, then remaining `\|` → `1`. Order of string replacements matters when one artifact is a substring of another. |
| 40 | **Use `max(kibble_counts)` not `first_clear` for the double-counting guard** | `first_clear` underestimates starting kibble because: (a) clear frames have 0 kibble (model only detects kibble with cats), (b) phase_entry fallback is measured while cat occludes some kibble. `max(kibble_counts)` = best camera angle with most kibble visible = closest to true bowl count. Also: always provide a fallback for `last_clear` via `_find_kibble_at_phase_exit`. |
| 41 | **Telegram notification first line must be the actionable verdict** | Push notifications show only the first ~100 chars. Burying the compensation verdict mid-message means the owner must open every message even when no action is needed. Put the verdict on line 1: `✅ No compensation needed` or `⚠️ Sanbo came at HH:MM — compensate Dan ~N kibble`. |
| 42 | **Episode numbers must be day-wide, not per-clip** | Each video clip creates its own `FeedingTracker`, so episode indices reset to 0 per clip. If clip 1 has 2 hand episodes and clip 2 has 1, the second clip's snapshot was also named `dan_hand_ep0` — indistinguishable from the first clip's. Fix: pass `episode_offset` (cumulative confirmed episodes from prior clips) to each new `FeedingTracker`. |
| 43 | **kibble_dispensed snapshot must be captured immediately after the hand episode, not after stability** | Waiting for 3 consecutive stable clear frames (no cat) to confirm kibble count means the snapshot is taken after Dan has been eating for minutes — bowl nearly empty. Dan goes straight to the bowl after hand feeding, so "clear" frames only appear when he leaves. Fix: take the first clear frame within 3 seconds of episode end; fallback to highest-kibble frame within 5 seconds if Dan stays at bowl the whole time. |
| 44 | **GitHub Actions cron time is not actual start time** | Scheduled morning-report runs can start hours after the configured cron. Compensate with an intentionally early cron, guard with a wait-until-feeding-window-complete step, and include scheduler heartbeat in Telegram + GitHub summaries so delay is separated from notebook runtime. |
| 45 | **Daily Roboflow flags must be trended, not just uploaded** | A one-day flag count helps monitor model drift, but monthly retraining needs history. Log `flagged_frames`, Roboflow uploaded/skipped/failed counts, and top flag tags to `feeding_log.csv` so retraining direction is based on recurring failure modes. |
| 46 | **Kibble-jump flags must ignore cat-occluded bowl frames** | During normal eating, cats occlude kibble and make visible kibble counts jump. Only compare clear bowl frames for `kibble-jump-*`; otherwise Roboflow gets flooded with normal eating frames. |
| 47 | **Bowl-position alerts belong in the 24/7 Pi recorder, not the morning report** | Camera misalignment is a live operational issue. The morning report only sees the analyzed window; `motion_recorder.py` has the continuous frame stream and should send Telegram after 10 minutes of missing/off-center bowl detection. |
| 48 | **YOLOv8n COCO has lowercase `bowl` class** | Do not assume the Pi needs a custom Fair Feeder model for bowl-position monitoring. `yolov8n.pt` includes class 45 `bowl`; use it first for the lightweight 24/7 camera-position alert and only move to a custom model if real Pi frames prove unreliable. |
| 41 | **Never hardcode external IDs or Magic Strings in source code** | Hardcoding the Telegram Group ID (`-1003701186018`) in `motion_recorder.py` bypassed environment configuration best practices. Even if the ID isn't a direct security key, committing it to GitHub leaks internal identifiers and requires a code edit to change environments. Always extract Magic Strings and IDs into `.env` loaded via `os.getenv()` or a secrets manager like Infisical. |
| 42 | **`matplotlib.dates` + `plot_date` produces Jan 01 X-axis regardless of actual dates** | Using `mdates.date2num()` + `ax.plot_date()` on Pi (matplotlib 3.10) consistently showed Jan 01 for all points despite correct float values. Root cause unclear (possible epoch mismatch). Fix: abandon `matplotlib.dates` entirely — use integer indices for X positions with `ax.set_xticks()` + `ax.set_xticklabels([d[5:] for d in all_dates])` for MM-DD string labels. Simple, reliable, no date math needed. |
